RDMA memory key

IB协议中对lkey和rkey（后面统称memory key）并没有规定每个厂商、驱动要如何管理和产生这个key。所以，实际上每个厂商对这个key如何产生如何管理并不统一。

当然，协议中并不是完全没有对这个key进行过描述和规定，相关的描述和规定会在后文中提及。

由于memory key的名字，很多人下意识认为这个key有秘钥的作用。那么它到底是否有秘钥或者说安全性的功能在里面呢？

memory key是否有安全性的作用呢？

个人理解,memory key并不能够承载秘钥的作用的，实际上在IB SPEC中对memory key进行了简单的定义，它是24 bit index，和8 bit key组成的，这8bit key要允许用户自由更改。

但是！有两个问题：

  1. 上面的这个定义是由前提的是当设备支持base memory management extension时才需要遵循以上的固定，也就是要支持MW等操作时才要求由8 bit key，否则这32 bit全都是index；
  2. key有切仅有8个bit，很难想象这8bit能提供多大的安全性，它甚至不能覆盖三位数的密码；

因此，从上面来看，memory key主要的作用就是index，而剩余的8bit key是另有他用的。

  • 这个index又是什么呢？
    它实际上就是让硬件查询硬件表项的一个索引，硬件通过该index，可以区分不同进程所使用的内存，并通过它找到对应的页表，从而完成读写内存的操作。

memory key中的key部分有什么用？

memory key中的key的作用在IB SPEC中并没有描述，仅仅是说明了这部分要求HCA允许用户任意修改。因此，我们要从应用的角度去看，这个key到底有什么作用。

用户态只有在MW中才有对key值修改的操作，具体体现就是ibv_inc_rkey(),他仅在MW中使用到。（当前仅有mlx和irdma支持MW，并且MW的实现都是通过post_send，整个行为类似于FRMR）

内核态则只有FRMR相关的操作会修改key值，ib_update_fast_reg_key()和ib_inc_rkey()；（通常修改rkey也仅仅使得rkey + 1，因此要说安全性基本上就是没有）

通过内核态以及用户态代码的分析，这个key的修改主要是用来做FRMR的。在FRMR的参数中，它允许用户修改key值。然后结合FRMR的使用场景，我们大概就可以推断这个key值的作用了。就是用来快速的使前一个MR失效。

通常，使用的场景如下：

  1. 用户通过invalid操作，使得一个MR迅速失效。
  2. 通过FRMR快速的将这个MR重新生效，换上新的内存（DMA buf）

如果，在上面的场景中，key值没有发生变化，则对端还可以通过老的key值访问这块新的内存，如果业务代码（对端）没有及时的停止对这个memory key的访问，那这将可能是个灾难。因此，FRMR允许用户修改这个key值，修改key值后老的memory key实际上就失效了。即使业务代码中存在漏洞，也不会引入严重的问题。

同时，由于能变化的只有key部分，index部分保持不变，因此，对于驱动和硬件来说，实际上使用到的“资源”还可以复用，也就是FRMR时，减少软硬件重新申请释放资源的开销。提高了FRMR的效率。

各个厂商的实现

下面列举下各个厂商对memory key的实现，一方面也是佐证上面的推断。

MLX

MLX的index部分是由固件获取的，key值默认不为零，由一个全局key值决定，每次新分配一个MR之后，这个key值加一。

irdma

irdma实际上对key的安全性做的非常高，它是唯一一家随机生成memory key的厂商。

它会生成一个随机数以及一个index值（这个index也是由这个随机数生成的，在一个bitmap中从随机值开始查找到第一个未使用的值即是index），可以说它的memory key基本上就是一个随机数。

它的memory key结构如下：
| 随机数高位| index(由随机数获取的)| 随机数低位 |

efa

efa的key值完全来自于固件.

erdma

erdma当前的key部分仅支持为0,index部分这是自己实现的bitmap分配的

mana

mana的key值完全来自于固件

RDMA操作的安全性如何实现？

对于send/recv这种语义，本来RDMA和传统的网络就没有太大区别。主要的疑问点在于write/read/atomic这类的操作，这类操作实际上是内存语义，支持两个主机之间“直接内存访问”。考虑到这一点，很难想象这里面没有安全或者加密的机制。但是，也很难相信通过一个8 bit的key就可以实现通信的安全。

现在，MLX已经支持了RoCE的IPSEC通信，这对RoCE的安全性来说是大大的提高了。另外，对于内存语义，如果MR相关的内存本身就实现了加密，则也可以实现内存操作的安全性，只不过这对性能可能有些影响。

ERDMA

基础使用

在阿里的RDMA被他们称为ERDMA，这里的弹性目前不知道指的是什么。我曾经通过代码分析，发现ERDMA上传的驱动中并没有
PF的驱动，因此我们怀疑它是VF。但是现在经过对mlx以及其他DPU的了解，或许它连VF都算不上。或者说是一种特殊的
VF，由智能网口直接虚拟出来的PCI设备。就像我们的平台设备可以通过一些硬件模块模拟PCIE设备（PCIE化）。一张
智能网口想要模拟成一个bridge，并在里面仿真出一些pcie设备简直易如反掌。

从他们的帮助文档来看，他们将ERDMA称为ERI(ERDMA interface?), RDMA网口和普通的网口是分离使用的，或者说RDMA网口虽然有个网络设备，但是流似乎也都是RDMA的。
虽然这么说，但是在ERI的创建中似乎又要求ERI依附于一个ENI(Elastic network interface?)。

而且似乎大多数应用在使用这个网口时就会直接使用RDMA，这里猜测可能和他们一直在主推的SMC有关系。
原本SMC是由redhat推出的技术，但是现在主要是阿里的人在投入维护扩展这个模块。
在他们对SMC的介绍中，可以看出来，SMC是可以做到透明替换TCP/IP的。（或许可以将他们的example下下来进行分析一波？）
https://help.aliyun.com/document_detail/327118.html

这里有两种猜测：

  1. 实际上这个网口就是一个虚拟的网络设备，他是由ERDMA设备创建的一个设备，然后流量直接由SMC导入了ERDMA。(这种猜测的原因是因为从代码来看ERDMA设备就是一个独立的PCI设备)
  2. ERDMA仍然是一个传统意义上的网络设备，然后他同时支持RDMA，在阿里的设置中，这个设备主要使用RDMA的功能。

linux dns setup

目的

这个文档是用来记录日常使用linux冲浪时遇到的与DNS相关的一些问题，以及一些相关的小技巧

修改DNS的坑

在使用ubuntu 18.04的时候，想要上下github，但是因为众所周之的原因，就很慢，甚至github.io根本就没法连接。于是想要通过修改dns和host的方式来加快一下浏览的速度（主要原因是不想去折腾科学上网）。但是，修改hosts文件根本无法生效，修改DNS也会在重启之后实效。于是不得已折腾了小半天这个DNS的设置。

在ubuntu（18）中，系统默认开启了systemd的resolved（后文对resolved有相关介绍），因此会默认将DNS设置为127.0.0.53，通常我们使用时，都只修改/etc/resolv.conf和network-script里面的dns的设置，这这种修改仅仅只会使得DNS的修改生效一段时间，在重启网络或者重启系统后，resolv.conf的设置就会又被resolved改回去。

所以正确的打开方式是：

  • 修改/etc/systemd/resolved.conf,里面有DNS以及fallbackDNS的设置，无论设置哪个都可以起到效果，不过建议修改fallback DNS，将我们需要添加的DNS放入fallback DNS中就可以了。以免修改了DNS影响resolved的功能（让resolved决定默认的DNS）。
  • 修改/etc/hosts，这个时候可以修改/etc/hosts了，将我们需要加速的网站写入/etc/hosts中
  • 对于一些启用了nscd的机器，可能还需要刷新一下DNS的缓存: systemctl restart nscd（service也行，系统用啥管理就用啥重启服务）

resolved和resolvectl

resolved是systemd的一部分，它为系统提供了DNS缓存还有LLMNR以及multicastDNS的功能。（实际上有很多商用OS并不会启用它，因此很多系统需要其他的DNS缓存服务例如nscd）。

在使用resolved时，resolvectl就是与resolved配套使用的工具，可以用来在线设置查询相关的DNS服务。

  • resolvectl status：查看当前resolved的状态，包括当前DNS运行的状态(这是目前指导的唯一的有效的方法查看当前DNS的手段)
  • resolvectl statistics:查看那当前的统计信息
  • resolvectl dns enp3s0 114.114.114.114手动设置DNS服务
  • resolvectl query https://hellotcc.github.io 查看当前的网站的DNS查询结果

nscd

与win不一样，默认linux下没有DNS缓存机制，需要自行添加相关的服务nscd，当然nscd也不仅仅提供DNS的缓存能力，还提供了passwd, groups，hosts, netgroup等功能的缓存服务

perf

perf的原理

perf list

perf list用来显示perf相关的事件有哪些，可以显示perf可以跟踪的事件符号。（这些事件都是可以在下面的命令中通过-e指定的事件）

注：对于非特权用户只支持少量的perf事件，通常只支持上下文切换方面的时间，其中包括少数的CPU的PMU事件和一些软件事件，其余的PMU和软件事件则不支持。（也允许通过sysctl设置kernel.perf_event_paranoid为-1，来为非root用户提权）

perf top

perf top的功能类似

perf stat

perf stat是获取性能counter统计数据的工具。其中的counter类型就是perf list中包含的event，其中包括硬件统计（PMU，performance monitor unit）也包括软件trace event（可以在ftrace系统中找到相关的信息和实现）的统计。

通常在对一个系统进行评估时，我们会定义一些指标，perf stat可以帮助我们从系统中获取原始的数据，我们可以通过计算来获得我们需要的这些指标的值，最终实现对一个“系统”的评估。

常用的参数：

  • -e，指定event，相关event可以通过上述的perf list获取
  • -p，指定需要profiling的进程的pid，多个pid之间通过逗号分割

topdown

Topdown本身是一种由IBM提出来的方法论，常本用于电子行业的设计领域。我们这里说的Topdown通常指的是针对CPU的微架构分析方法Top-down Microarchitecutre Analysis Method（a.k.a TMAM）。

TMAM将Topdown作为一种方法论，引入到了当前CPU的流水线运行瓶颈的分析之中，将流水线的模型抽象成了四个部分（front-end bound,back-end bound, bad speculation和retiring）。这样一个抽象模型为用户打开了一个统一的全局视角，实现了跨架构的分析方法。同时也去掉了大量不必要的细节，减少用户的学习成本（因为传统的分析方法，对于用户来说需要大量的cpu架构相关的知识，这个学习成本极高，并且用来分析的数据成千上万，无法直接使用，TMAM在这里定义了一些重要的性能指标）。通过典型的指标分析到软件相应的瓶颈后，用户可以通过针对性的优化手段来提高软件在当前CPU微架构下的性能。

支持TMAM的工具有不少，包括intel的Vtune和perf stat。当然要支持TMAM也依赖与驱动和硬件本身，intel 是在sandy bridge之后加入了对TMAM的支持，arm架构则更加混乱一些，虽然定义了标准的事件，但是最终还是依赖各个厂商的实现情况，如果相关产家没有实现标准的一些PMU事件，则可能需要通过一些其他的方法来支持TMAM的分析了（这些非标的设计可能提高了分析的精度、提供了更多的信息或者降低了硬件的成本，但是不得不说，这都是在提高用户的学习成本）。

这里简单总结一下TMAM中各个环节的特点：

  1. front-end
    Front end包括下一条指令地址的预测、取指、译码、微码的转换这些内容。也就是cpu执行的最前端，通常有两个指标fetch latency和fetch bandwidth。

    Fetch latency指没有指令被发射的周期。导致这个问题主要有icache和iTLB的miss，除此之外还有一项，即分支预测导致的再次取指的过程(branch resteers)被当作fetch delay计算到了front end中。(PS:分支预测导致的pipeline flush的部分则是被算到了bad speculation的Machine clear中，即这两项存在一定的耦合关系)

    Fecth bandwidth指发射不满的周期，比如可以四发射的cpu，不满四发射的周期的数量。它可能指向了译码效率低的问题。

    整体来看front end成为瓶颈点的可能性比较小，但是也不是不可能的，比如在解释性语言中分支预测的效率可能比较低，这一类的优化可能需要从编译器或者解释器入手。（这里体现出了c等编译型语言的性能优势点之一了，可以获得更好的BP效果）

  2. back-end
    back-end负责指令的发射，它会监控当前微码所需的数据是否准备完毕，是否满足发射条件，并根据实际情况将微码发射到发射队列中（这个过程被称为allocate）。因此backend统计通常反映了指令未能有效的发射出去的问题（现在的cpu通常都支持多发射，有多个将微码发送到执行单元的队列，backend反映的就是这些发射队列中的气泡的问题）。最常见的导致这类问题的原因就是Data cache miss。通常backend根据问题的原因又可以分为：Memory bound 和 core bound。

    Memory bound的问题是最常见的，指的是由于内存子系统或者cache引入的cpu stall。该类别问题的子项很多，例如各个level的cache miss，由于内存序导致的回写，甚至也有可能并没有发生cache miss而是由于cache的一致性（同步）引入了延迟从而导致了stall（这个场景下，一个数据存在多个cache中，由于需要确定数据的合法性，相比与没有这个过程来说增加了一定的延迟）。 这种问题

    Core bound的问题通常比较难以界定的，通常是由于执行单元的限制（多个指令需要同一个执行单元执行,即计算资源不足够）或者代码的并行度不够造成的（这里指的并不是通常意义的并行执行，而是指代码前后之间存在依赖关系，无法实现乱序执行）。这一类的问题通常也没法在代码中进行优化，需要优化cpu的设计或者是优化编译。

  • 分割线（通常把上面两种称为前段，后面两种称为后端）

  1. bad speculation
    bad speculation主要指由于错误的投机执行导致的pipeline slot的浪费，其中包括了在执行单元中被丢弃从而未能retire的指令（微码）以及由早期错误的投机执行导致的recovery操作引发的发射队列阻塞导致的slot浪费。
    大体上bad speculation又被分为了两类，branch mispredict和Machine clear。其中branch mispredict包括分支预测还有错误的投机读。然而由于分支预测导致的pipeline flush的部分则被算到了Machine clear中，除此之外中断异常等引发的pipeline flush也都是Machine clear的一部分。

  2. retiring
    retiring就是最终成功执行的指令，代码优化的最终目的就是提高该项指标，如果是100%则说明代码已经没有进一步优化的空间了
